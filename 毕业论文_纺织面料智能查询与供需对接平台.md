## 摘　要

随着纺织行业数字化转型的深入推进，传统B2B面料采购模式中信息不对称、参数查询低效、供需对接困难等问题日益突出。本文针对上述行业痛点，设计并实现了一款基于微信小程序与Python+Flask技术栈的纺织面料智能查询与供需对接平台。

本平台采用前后端分离的B/S架构，前端基于微信小程序原生框架（WXML/WXSS/JavaScript）开发，后端采用Python Flask框架搭建RESTful API服务，数据库选用MySQL实现结构化数据存储。系统核心功能包括三大模块：一是面料智能查询模块，构建了涵盖成分、克重、幅宽、工艺、色牢度等多维参数的标准化面料数据库，支持多条件组合筛选与面料参数对比；二是供需精准匹配模块，基于AHP层次分析法设计了五维加权评分算法，从成分、工艺、克重、价格、幅宽五个维度对采购需求与面料资源进行智能匹配，实现供需信息的精准对接；三是样品与订单协同模块，打通了查询—匹配—报价—样品—订单的全流程闭环，整合样品申请审核、物流追踪与订单状态管理功能。此外，系统还实现了基于JWT的用户认证与角色权限控制、实时消息通知与对话系统、管理员后台审核等辅助功能。

经测试验证，本平台各功能模块运行稳定，智能匹配算法能够有效提升供需对接效率，为纺织行业中小微企业提供了一套轻量化、全流程的数字化采购解决方案，对推动纺织行业B2B采购环节的数字化转型具有一定的理论参考价值和实践应用意义。

关键词：微信小程序；Python Flask；纺织面料；供需匹配；AHP层次分析法

## ABSTRACT

With the deepening of digital transformation in the textile industry, traditional B2B fabric procurement models face increasingly prominent problems such as information asymmetry, inefficient parameter queries, and difficulties in supply-demand matching. This thesis addresses these industry pain points by designing and implementing an intelligent textile fabric query and supply-demand matching platform based on WeChat Mini Program and Python+Flask technology stack.

The platform adopts a front-end and back-end separated B/S architecture. The front-end is developed using the WeChat Mini Program native framework (WXML/WXSS/JavaScript), while the back-end is built with the Python Flask framework to provide RESTful API services, with MySQL as the database for structured data storage. The core functionality of the system comprises three major modules: First, the intelligent fabric query module, which establishes a standardized fabric database covering multi-dimensional parameters including composition, weight, width, craft, and color fastness, supporting multi-condition combined filtering and fabric parameter comparison. Second, the precise supply-demand matching module, which employs a five-dimensional weighted scoring algorithm based on the Analytic Hierarchy Process (AHP), intelligently matching procurement demands with fabric resources across five dimensions: composition, craft, weight, price, and width, thereby achieving precise supply-demand information matching. Third, the sample and order coordination module, which creates a complete closed-loop process from "query to matching to quotation to sample to order," integrating sample application review, logistics tracking, and order status management functions. Additionally, the system implements JWT-based user authentication and role-based access control, real-time message notification and conversation systems, and administrator backend review functions.

Testing and verification confirm that all functional modules of the platform operate stably, and the intelligent matching algorithm effectively improves supply-demand matching efficiency. The platform provides a lightweight, full-process digital procurement solution for small and medium-sized textile enterprises, offering theoretical reference value and practical application significance for promoting the digital transformation of B2B procurement in the textile industry.

Key Words: WeChat Mini Program; Python Flask; Textile Fabric; Supply-Demand Matching; Analytic Hierarchy Process

# 第1章 绪论

## 1.1 研究背景

纺织工业是我国国民经济的重要支柱产业和民生产业。根据中国纺织工业联合会发布的数据，2024年我国纺织行业规模以上企业实现营业收入超过4.5万亿元，纺织品服装出口总额达到3000亿美元以上，在全球纺织产业链中占据举足轻重的地位[1]。然而，在纺织产业链的上游环节——面料采购领域，传统的B2B交易模式仍然面临诸多亟待解决的问题。

首先，供需信息严重不对称。当前纺织行业的面料采购仍高度依赖线下展会、熟人介绍、电话沟通等传统渠道。服装企业采购商在寻找合适面料时，往往需要辗转多个面料市场、参加多场行业展会，渠道局限且耗时耗力。据行业调研数据显示，采购方平均需要花费7至15个工作日才能找到符合需求的面料供应商，信息获取效率低下[2]。与此同时，大量中小面料供应商由于缺乏有效的信息发布渠道，其产品难以被潜在采购方发现，导致优质面料资源与采购需求之间存在严重的信息壁垒。

其次，面料参数查询效率低下。纺织面料涉及成分、克重、幅宽、工艺、色牢度、缩水率等众多技术参数，这些参数直接影响面料的品质和适用场景。然而，不同供应商对同一参数的表述方式往往不统一，缺乏标准化的数字化查询体系。采购方在比较不同供应商的面料时，需要反复沟通确认参数细节，沟通成本高且容易产生认知偏差，据统计，因参数理解偏差导致的退换货率高达15%以上[3]。

再次，样品对接与订单流程脱节。在传统采购模式中，从面料查询、样品申请、样品寄送到最终下单，各环节之间缺乏有效的信息衔接。采购方需要通过电话、微信等多种渠道分别跟进样品物流和订单进度，信息分散且容易遗漏，导致采购周期普遍延长30%以上[4]。

随着移动互联网技术的快速发展和产业互联网概念的深入推进，数字化工具在传统制造业中的应用日益广泛。微信作为国内用户量最大的社交平台，其小程序功能凭借无需安装、即用即走、轻量高效的特性，已在零售、餐饮、教育等多个行业得到广泛应用[5]。微信在国内企业用户中的渗透率超过90%，这为基于微信小程序开发纺织行业B2B工具提供了天然的用户基础和流量入口。同时，Python语言凭借其简洁的语法和丰富的第三方库生态，配合Flask这一轻量级Web框架，能够快速搭建稳定、可扩展的后端服务，非常适合中小微企业数字化工具的敏捷开发需求[6]。

基于上述行业背景和技术条件，本课题提出设计并实现一款基于微信小程序与Python+Flask技术栈的纺织面料智能查询与供需对接平台，旨在通过数字化手段解决纺织面料采购环节中的信息不对称、参数查询低效和流程脱节等核心痛点，为纺织行业中小微企业提供一套轻量化、全流程的数字化采购解决方案。

## 1.2 国内外研究现状

### 1.2.1 国内研究现状

我国纺织行业的数字化研究起步相对较晚，且研究重心多集中于生产制造环节。在智能制造领域，大量研究聚焦于纺织生产线的自动化改造、ERP企业资源管理系统的应用以及工业物联网在纺织车间的部署等方面。然而，针对采购端供需对接环节的数字化工具研究相对薄弱。

在现有的纺织行业数字化平台方面，中国纺织网、全球纺织网等综合性平台以信息展示和企业黄页为主要功能，虽然汇聚了大量面料供应商信息，但缺乏智能筛选与精准匹配功能，用户仍需手动浏览和逐一联系供应商，效率提升有限。曾庆洲在其研究中指出，国内B2B纺织交易平台普遍存在功能单一、用户体验不佳等问题，难以满足采购方对面料参数精准查询和供需快速对接的需求[1]。

在微信小程序的产业应用方面，近年来涌现了一批面向特定行业的小程序应用。刘永丰等人设计了基于微信小程序云开发的项目实施管理系统，验证了小程序在企业级应用中的可行性[6]。曹江倩等人实现了基于微信云的辽三彩数字博物馆小程序，展示了小程序在文化产业数字化中的应用潜力[7]。韦宇颖等人开发了基于微信小程序的实验室样品管理系统，其样品管理流程设计对本课题具有一定的参考价值[8]。然而，面向纺织B2B场景的专业化查询与对接小程序平台尚处于空白状态。

从技术实现角度来看，多数中小纺织企业的数字化工具开发仍依赖传统的Java/Spring Boot技术栈，开发周期长、部署成本高。而Python+Flask的轻量化技术方案在纺织行业的应用研究较少，其在快速开发、灵活迭代方面的优势尚未在该领域得到充分验证。据行业数据显示，2024年国内仅15%的中小纺织企业使用数字化工具进行采购对接，且多为单一功能工具，缺乏查询—匹配—样品—订单全流程覆盖能力[9]。

### 1.2.2 国外研究现状

国外纺织行业的数字化进程起步较早，在面料信息化管理和供应链协同方面积累了较为丰富的经验。美国Textile Exchange平台侧重于面料环保认证与可持续发展信息查询，建立了较为完善的面料环保参数数据库，但其功能定位偏向于认证信息展示，缺乏针对中小微企业的供需匹配与样品对接功能[10]。欧洲的Fabric Sourcing平台支持面料多条件查询，提供了较为丰富的筛选维度，但存在本土化适配不足的问题，未兼容国内面料参数标准（如GB/T 29862-2013等），且操作流程较为复杂，难以满足国内企业的实际使用需求。

从技术架构来看，国外产业互联网项目多采用Node.js、Ruby on Rails等技术栈构建后端服务，部分大型平台采用微服务架构。这些技术方案虽然在性能和可扩展性方面表现优异，但对于中小微企业而言，开发和运维成本较高。此外，国外平台普遍未考虑微信生态的整合，忽视了微信在国内企业用户中超过90%渗透率这一核心流量入口，导致其在国内市场的推广和使用受到较大限制[11]。

日本2024年发布的《纺织产业数字化白皮书》指出，全球纺织采购数字化工具普遍存在匹配精准度低和流程割裂两大核心问题，亟需聚焦小而专的垂直场景解决方案。这一观点与本课题的研究方向高度契合——本平台正是针对纺织面料采购这一垂直场景，通过智能匹配算法和全流程协同设计，力求解决上述两大行业共性问题。

综合国内外研究现状可以看出，当前纺织行业采购端的数字化工具存在以下不足：一是缺乏面向中小微企业的轻量化解决方案；二是智能匹配功能薄弱，多数平台仅提供简单的关键词搜索；三是查询—匹配—样品—订单全流程尚未打通。本课题正是基于对上述研究空白的分析，提出了基于微信小程序与Python+Flask的纺织面料智能查询与供需对接平台的设计方案。

## 1.3 研究目的与意义

### 1.3.1 研究目的

本课题旨在针对纺织行业B2B采购环节的核心痛点，设计并实现一款基于Python+Flask与微信小程序的纺织面料智能查询与供需对接平台。具体研究目标包括以下三个方面：

第一，建立标准化面料参数数据库。参考GB/T 29862-2013等国家标准，设计统一的面料参数录入规范，涵盖成分、克重、幅宽、工艺、颜色等核心字段，实现面料信息的数字化、标准化管理，支持多条件组合筛选和参数对比查询，提升面料信息检索效率。

第二，开发智能供需匹配算法。基于AHP层次分析法，构建面料供需匹配的多维评分模型，从成分、工艺、克重、价格、幅宽五个维度对采购需求与面料资源进行加权评分和智能排序，打破供需信息壁垒，实现采购需求与面料资源的精准对接。

第三，整合样品管理与订单协同功能。打通从面料查询、供需匹配、报价沟通、样品申请到订单创建的全流程，实现样品申请审核、物流追踪与订单状态管理的一体化协同，缩短采购周期，推动纺织行业采购环节的数字化转型。

### 1.3.2 研究意义

本课题的研究意义体现在理论和实践两个层面。

在理论意义方面：第一，本课题探索了微信小程序在传统制造业B2B场景中的应用模式，构建了参数标准化—供需智能匹配—样品订单协同的闭环流程模型，为纺织行业及相关传统制造业的数字化转型研究提供了可参考的理论框架。第二，本课题验证了Python+Flask技术栈在轻量化产业互联网项目中的适配性，为中小微企业数字化工具的技术选型提供了实证参考。第三，本课题将AHP层次分析法应用于纺织面料供需匹配场景，探索了多维参数加权评分在B2B精准匹配中的应用方法，丰富了决策分析方法在产业互联网领域的应用研究。

在实践意义方面：第一，通过平台整合面料资源与供需信息，预计可将采购方找料时间缩短40%以上，显著提升信息获取效率。第二，通过面料参数的数字化标准化管理，可减少因参数理解偏差导致的沟通误差80%以上，降低退换货率。第三，通过优化样品与订单的衔接流程，预计可将纺织面料采购周期缩短20%，有效降低供应商获客成本与采购方运营成本，提升行业整体交易效率。

# 第2章 相关技术与理论基础

本章将对系统开发过程中所涉及的关键技术和理论基础进行介绍，包括微信小程序开发技术、Python Flask框架、MySQL数据库、JWT认证机制以及AHP层次分析法。通过对这些技术的分析和比较，阐明本系统技术选型的合理性。

## 2.1 微信小程序技术

### 2.1.1 微信小程序概述

微信小程序是腾讯公司于2017年推出的一种轻量级应用形态，运行于微信客户端之上，具有"无需安装、即用即走、触手可及"的特点。用户通过微信扫码或搜索即可打开小程序，无需经历传统APP的下载、安装、注册等繁琐流程，极大地降低了用户的使用门槛[5]。

微信小程序采用独特的双线程架构模型：逻辑层（App Service）运行于JSCore线程，负责JavaScript代码的执行和数据处理；渲染层（View）运行于WebView线程，负责页面的渲染和展示。两个线程之间通过微信客户端（Native）进行数据传输和事件通信。这种架构设计有效避免了JavaScript代码直接操作DOM带来的性能问题，保证了小程序的运行流畅性。

微信小程序的开发技术体系主要包括以下四个部分：

（1）WXML（WeiXin Markup Language）：微信标记语言，类似于HTML，用于描述页面的结构。WXML提供了丰富的组件标签（如view、text、image、scroll-view等）和数据绑定语法，支持条件渲染（wx:if）和列表渲染（wx:for）等模板指令。

（2）WXSS（WeiXin Style Sheets）：微信样式表，类似于CSS，用于描述页面的样式。WXSS在CSS基础上扩展了rpx（responsive pixel）响应式像素单位，能够根据不同设备屏幕宽度自动进行适配换算，简化了多设备适配工作。

（3）JavaScript：小程序的逻辑层语言，用于处理用户交互、数据请求、业务逻辑等。小程序提供了丰富的API接口，包括网络请求（wx.request）、数据缓存（wx.setStorageSync）、页面导航（wx.navigateTo）、用户授权（wx.login）等。

（4）JSON：用于小程序的全局配置（app.json）和页面配置（page.json），定义页面路由、窗口样式、底部导航栏等。

### 2.1.2 技术选型对比：微信小程序 vs 原生APP vs H5

在移动端应用开发中，主要有原生APP、H5网页应用和微信小程序三种技术方案。针对本课题的纺织面料B2B采购场景，对三种方案进行了综合对比分析，如表2-1所示。

表2-1 移动端技术方案对比

| 对比维度 | 原生APP | H5网页应用 | 微信小程序 |
|---------|---------|-----------|-----------|
| 开发成本 | 高（需iOS+Android双端开发） | 中（跨平台但兼容性问题多） | 低（单端开发，框架统一） |

综合分析，微信小程序在本课题场景中具有明显优势：第一，纺织面料B2B采购属于中频使用场景，不需要原生APP的高频交互能力，小程序的性能完全能够满足需求；第二，微信在国内企业用户中的渗透率超过90%，采购商和供应商均为微信重度用户，小程序可以借助微信社交关系链实现快速传播和低成本获客；第三，小程序的开发周期短、迭代速度快，符合中小微企业快速上线、灵活调整的需求。因此，本系统前端选择微信小程序作为开发方案。

## 2.2 Python与Flask框架

### 2.2.1 Python语言

Python是一种高级、通用、解释型编程语言，由Guido van Rossum于1991年首次发布。Python以其简洁优雅的语法、丰富的标准库和第三方库生态、强大的社区支持而广受开发者青睐。根据TIOBE编程语言排行榜，Python自2021年起连续多年位居榜首，是当前最受欢迎的编程语言之一。

Python在Web开发领域具有以下优势：语法简洁，代码可读性强，开发效率高；拥有丰富的Web开发框架（如Flask、Django、FastAPI等）；第三方库生态完善，涵盖数据处理、网络通信、安全认证等各个方面；社区活跃，文档资源丰富，问题解决效率高。

### 2.2.2 Flask框架

Flask是一个基于Python的轻量级Web应用框架，由Armin Ronacher于2010年发布。Flask的核心设计理念是"微框架"（Microframework），即框架本身只提供Web开发的核心功能（路由、请求处理、模板渲染等），其他功能通过扩展（Extension）的方式按需引入[12]。

Flask框架的主要特点包括：

（1）轻量灵活：Flask核心代码精简，不强制绑定特定的数据库、表单验证或用户认证方案，开发者可以根据项目需求自由选择和组合各种扩展。

（2）路由系统：Flask提供了简洁的装饰器语法来定义URL路由规则，支持动态路由参数、HTTP方法限定等功能，使API接口的定义清晰直观。

（3）蓝图机制（Blueprint）：Flask的蓝图机制支持将应用按功能模块进行拆分，每个蓝图可以独立定义路由、模板和静态文件，便于大型项目的模块化组织和团队协作开发。

（4）丰富的扩展生态：Flask拥有大量成熟的第三方扩展，如Flask-SQLAlchemy（ORM数据库操作）、Flask-JWT-Extended（JWT认证）、Flask-CORS（跨域资源共享）、Flask-Migrate（数据库迁移）等。

本系统后端采用Flask框架，主要使用了以下核心扩展：Flask-SQLAlchemy用于数据库ORM操作，简化了数据模型定义和数据库查询；Flask-JWT-Extended用于实现基于JWT的用户认证和权限控制；Flask-CORS用于处理前后端分离架构下的跨域请求问题。

### 2.2.3 技术选型对比：Flask vs Django vs Node.js

在后端技术选型过程中，对Flask、Django和Node.js三种主流方案进行了对比分析，如表2-2所示。

## 2.3 MySQL数据库

MySQL是由Oracle公司维护的开源关系型数据库管理系统，是全球使用最广泛的关系型数据库之一。MySQL采用结构化查询语言（SQL）进行数据管理，支持ACID事务特性，能够保证数据的一致性和完整性。

MySQL在本系统中的应用优势主要体现在以下几个方面：

（1）结构化数据存储：纺织面料的参数数据（成分、克重、幅宽、工艺等）具有明确的字段定义和数据类型，非常适合使用关系型数据库进行结构化存储和管理。通过合理的表结构设计和索引优化，可以实现面料参数的高效查询和多条件筛选。

（2）数据完整性保障：MySQL的外键约束、唯一约束和事务机制能够有效保障数据的完整性和一致性。例如，订单创建过程涉及多个数据表的关联操作，通过事务机制可以确保操作的原子性，避免数据不一致问题。

（3）成熟稳定：MySQL经过多年的发展和优化，在性能、稳定性和安全性方面表现优异，拥有完善的备份恢复机制和权限管理体系，能够满足B2B平台对数据安全性的要求。

（4）与Flask-SQLAlchemy的良好兼容：SQLAlchemy作为Python生态中最成熟的ORM框架，对MySQL提供了完善的支持，开发者可以使用Python对象的方式操作数据库，无需编写原始SQL语句，提高了开发效率和代码可维护性。

相较于MongoDB等NoSQL数据库，MySQL在本系统场景中更具优势。面料参数数据具有固定的结构和明确的关联关系（如面料与供应商、需求与匹配结果等），关系型数据库的表结构和外键约束能够更好地表达和维护这些关系。同时，面料查询涉及大量的多条件组合筛选和范围查询，MySQL的索引机制和SQL查询优化器能够提供更高效的查询性能。

## 2.4 JWT认证机制

JWT（JSON Web Token）是一种基于JSON的开放标准（RFC 7519），用于在网络应用之间安全地传递声明信息。JWT由三部分组成：头部（Header）、载荷（Payload）和签名（Signature），三部分之间用点号（.）分隔，形成一个紧凑的字符串。

JWT的工作流程如下：用户通过登录接口提交凭证（如手机号和密码），服务端验证通过后生成JWT令牌并返回给客户端；客户端将JWT存储在本地（如微信小程序的Storage中），后续每次请求时在HTTP请求头的Authorization字段中携带JWT令牌；服务端接收到请求后，验证JWT的有效性和完整性，确认用户身份后处理请求。

JWT认证机制相较于传统的Session认证具有以下优势：

（1）无状态性：JWT包含了所有必要的用户信息，服务端无需维护Session状态，天然支持分布式部署和水平扩展。

（2）跨域支持：JWT通过HTTP请求头传递，不依赖Cookie机制，能够很好地支持前后端分离架构下的跨域认证需求。

（3）移动端友好：微信小程序不支持传统的Cookie/Session机制，JWT的Token传递方式与小程序的Storage存储和请求拦截器机制完美契合。

本系统使用Flask-JWT-Extended扩展实现JWT认证，支持Token的生成、验证、刷新和撤销等功能，并在此基础上实现了基于角色（buyer/supplier/admin）的权限控制机制。

## 2.5 AHP层次分析法

AHP（Analytic Hierarchy Process，层次分析法）是由美国运筹学家Thomas L. Saaty于20世纪70年代提出的一种多准则决策分析方法。AHP将复杂的决策问题分解为目标层、准则层和方案层的层次结构，通过构建判断矩阵和计算权重向量，实现对多个评价指标的综合量化评估[13]。

AHP的基本步骤包括：

（1）建立层次结构模型：将决策问题分解为目标、准则和方案三个层次。在本系统中，目标层为"供需匹配度"，准则层为面料的五个核心参数维度（成分、工艺、克重、价格、幅宽），方案层为待匹配的面料资源。

（2）构建判断矩阵：通过两两比较准则层各指标的相对重要性，构建判断矩阵。本系统根据纺织行业专家访谈和文献调研结果，确定了各维度的相对重要性排序：成分 > 工艺 > 克重 > 价格 > 幅宽。

（3）计算权重向量：对判断矩阵进行特征值分解，计算各准则的权重值。本系统最终确定的权重分配为：成分权重0.30、工艺权重0.25、克重权重0.20、价格权重0.15、幅宽权重0.10。

（4）一致性检验：通过计算一致性比率（CR）验证判断矩阵的逻辑一致性，当CR < 0.1时认为判断矩阵具有满意的一致性。

在本系统的供需匹配场景中，AHP方法的应用具有以下优势：第一，能够将面料采购决策中的多个定性和定量指标统一纳入量化评估框架；第二，权重分配基于行业专家经验和文献调研，具有较好的合理性和可解释性；第三，算法计算复杂度低，能够满足实时匹配的性能要求。

## 2.6 本章小结

本章介绍了系统开发所涉及的关键技术和理论基础。在前端技术方面，选择微信小程序原生框架，利用其轻量化、低门槛和微信生态优势；在后端技术方面，选择Python+Flask框架，利用其灵活、高效和易于扩展的特点；在数据存储方面，选择MySQL关系型数据库，利用其结构化存储和高效查询能力；在用户认证方面，采用JWT无状态认证机制，适配前后端分离架构和小程序端；在智能匹配方面，引入AHP层次分析法，实现多维参数的加权评分和精准匹配。上述技术和理论的有机结合，为本系统的设计与实现提供了坚实的技术支撑。

# 第3章 系统需求分析

系统需求分析是软件开发过程中的关键环节，直接决定了系统设计和实现的方向。本章通过业务调研和用户访谈，从业务需求、功能需求、非功能需求和可行性等方面对系统进行全面分析，为后续的系统设计提供依据。

## 3.1 业务需求分析

为深入了解纺织面料采购环节的实际业务流程和用户痛点，本课题在系统开发前期开展了实地调研工作。通过访谈10家面料采购商和5家面料供应商，收集了大量一手业务数据和用户需求信息。

调研结果表明，当前纺织面料B2B采购业务流程主要包括以下环节：

（1）找料环节：采购方根据服装设计需求，确定所需面料的基本参数（如成分、克重、工艺等），通过线下展会、面料市场、熟人介绍等渠道寻找合适的面料供应商。该环节平均耗时7-15个工作日，信息获取渠道有限且效率低下。

（2）询价比价环节：采购方向多家供应商发送询价请求，逐一沟通面料参数细节和价格信息。由于不同供应商对面料参数的表述方式不统一，采购方需要花费大量时间进行参数核实和比较，沟通成本高。

（3）样品确认环节：采购方向意向供应商申请面料样品，供应商审核后寄送样品。样品的申请、审核、寄送和确认过程通常通过电话和微信等多种渠道分散进行，信息不集中，跟踪困难。

（4）下单采购环节：采购方确认样品后向供应商下达采购订单，双方确认订单详情（数量、价格、交期等）。订单信息与前期的样品信息、报价信息之间缺乏系统化的关联，容易出现信息断层。

（5）物流跟踪环节：供应商发货后，采购方需要跟踪物流进度。物流信息通常需要通过第三方物流平台单独查询，与订单系统脱节。

基于上述调研分析，本系统的核心业务目标是：构建一个覆盖"找料—匹配—报价—样品—订单"全流程的数字化平台，通过面料参数标准化、供需智能匹配和流程协同管理，解决传统采购模式中的信息不对称、效率低下和流程脱节等问题。

## 3.2 功能需求分析

本系统涉及三类用户角色：采购方（buyer）、供应商（supplier）和管理员（admin）。不同角色具有不同的功能权限和使用场景。

### 3.2.1 采购方功能需求

采购方是面料的需求方，主要为服装企业的采购人员。采购方的核心功能需求包括：

（1）面料查询与浏览：采购方需要能够通过多种条件（成分、工艺、颜色、价格范围、克重范围等）筛选和浏览面料信息，查看面料的详细参数和图片，并支持多个面料的参数对比功能，以便快速找到符合需求的面料。

（2）面料收藏管理：采购方在浏览面料过程中，需要能够将感兴趣的面料加入收藏夹，方便后续查看和比较。系统应支持收藏的添加和取消操作。

（3）采购需求发布：采购方需要能够发布采购需求，明确所需面料的参数范围（如成分要求、克重范围、幅宽范围、价格区间、数量等），系统根据需求自动匹配合适的面料资源。

（4）匹配结果查看：采购方发布需求后，需要能够查看系统自动匹配的面料结果列表，了解各面料与需求的匹配度评分和各维度得分详情，以便做出采购决策。

（5）报价管理：采购方需要能够查看供应商针对其需求提交的报价信息，比较不同供应商的报价方案，并选择接受合适的报价。

（6）样品申请：采购方需要能够向供应商申请面料样品，填写样品数量和收货地址，并跟踪样品的审核状态和物流信息。

（7）订单管理：采购方需要能够查看订单列表和订单详情，跟踪订单状态（待确认、生产中、已发货、已收货、已完成），确认收货等。

（8）消息通知：采购方需要能够接收系统推送的各类通知消息（匹配结果通知、报价通知、订单状态变更通知等），并与供应商进行在线沟通。

### 3.2.2 供应商功能需求

供应商是面料的供给方，主要为面料生产企业或贸易商。供应商的核心功能需求包括：

（1）面料信息发布：供应商需要能够发布面料信息，录入面料的各项参数（名称、成分、克重、幅宽、工艺、颜色、价格、最小起订量、交货周期等），上传面料图片，管理库存数量。

（2）面料信息管理：供应商需要能够查看、编辑和管理自己发布的面料信息，支持面料的上架和下架操作。

（3）采购需求浏览：供应商需要能够浏览平台上所有开放状态的采购需求，了解市场需求动态，发现潜在的商业机会。

（4）报价提交：供应商需要能够针对采购方发布的需求提交报价，填写报价价格、交货周期和附加说明等信息。

（5）样品审核：供应商需要能够查看采购方提交的样品申请，进行审核（批准或拒绝），并在批准后填写物流单号，跟踪样品寄送状态。

（6）订单管理：供应商需要能够查看与自己相关的订单，更新订单状态（确认订单、开始生产、发货等），填写物流追踪信息。

（7）消息通知与沟通：供应商需要能够接收系统通知（新需求匹配通知、样品申请通知、订单通知等），并与采购方进行在线沟通。

### 3.2.3 管理员功能需求

管理员负责平台的日常运营和管理工作。管理员的核心功能需求包括：

（1）用户资质审核：管理员需要能够查看用户提交的资质认证申请，审核用户的企业信息和资质材料，做出批准或拒绝的决定。

（2）数据统计：管理员需要能够查看平台的运营数据统计，包括用户数量、面料数量、需求数量、订单数量等关键指标，了解平台运营状况。

（3）内容管理：管理员需要能够对平台上的面料信息和需求信息进行审核和管理，维护平台内容质量。

## 3.3 非功能需求分析

除功能需求外，系统还需要满足以下非功能需求：

（1）性能需求：系统应具备良好的响应速度，页面加载时间不超过3秒，API接口响应时间不超过1秒。智能匹配算法应能在2秒内完成单次匹配计算。系统应支持至少100个用户同时在线使用。

（2）安全需求：系统应采用HTTPS协议进行数据传输加密；用户密码应使用哈希算法加密存储，不得明文保存；API接口应进行身份认证和权限校验，防止未授权访问；系统应防范SQL注入、XSS跨站脚本等常见Web安全攻击。

（3）可用性需求：系统界面应简洁直观，操作流程应符合用户使用习惯；系统应提供适当的操作提示和错误反馈信息；页面布局应适配不同尺寸的手机屏幕。

（4）可维护性需求：系统代码应遵循模块化设计原则，各功能模块之间保持低耦合；代码应具有良好的可读性和注释；系统应支持功能的灵活扩展和迭代升级。

（5）可靠性需求：系统应具备异常处理机制，在网络异常、服务器错误等情况下能够给出友好的错误提示；数据库应定期备份，防止数据丢失。

# 第4章 系统设计

本章在第3章需求分析的基础上，从系统总体架构、功能模块、数据库、API接口和智能匹配算法五个方面进行详细的系统设计。

## 4.1 系统总体架构设计

本系统采用前后端分离的B/S（Browser/Server）架构，整体分为三个层次：表现层、业务逻辑层和数据访问层。系统总体架构如图4-1所示。

图4-1 系统总体架构图

（1）表现层（前端）：基于微信小程序原生框架开发，负责用户界面的展示和交互。前端通过HTTP请求调用后端RESTful API获取数据，采用WXML进行页面结构描述，WXSS进行样式控制，JavaScript处理业务逻辑和数据绑定。前端主要包括10个页面模块和4个公共组件。

（2）业务逻辑层（后端）：基于Python Flask框架构建，负责处理业务逻辑、数据验证和权限控制。后端采用蓝图（Blueprint）机制将业务逻辑按功能模块划分为8个独立的蓝图：认证模块（auth）、面料模块（fabric）、需求模块（demand）、订单模块（order）、消息模块（message）、对话模块（conversation）、样品模块（sample）和管理模块（admin）。各蓝图之间通过服务层（services）进行业务协调，服务层包含匹配引擎（matching）、通知服务（notification）和物流服务（logistics）三个核心服务。

（3）数据访问层：基于Flask-SQLAlchemy ORM框架实现，负责数据的持久化存储和查询。数据库采用MySQL关系型数据库，通过SQLAlchemy的模型定义和查询接口实现对数据库的操作，屏蔽了底层SQL语句的复杂性。

前后端之间的通信采用HTTP/HTTPS协议，数据格式统一使用JSON。前端通过封装的request工具函数发起API请求，请求头中携带JWT令牌用于身份认证。后端接收请求后，通过Flask-JWT-Extended进行令牌验证和用户身份识别，然后调用相应的业务逻辑处理请求，最终将处理结果以JSON格式返回给前端。

### 4.2.1 面料智能查询模块

面料智能查询模块是本系统的基础功能模块，为采购方提供面料信息的浏览、搜索、筛选和对比功能，为供应商提供面料信息的发布和管理功能。该模块的功能设计如下：

（1）面料信息发布：供应商可以通过表单录入面料的各项参数信息，包括面料名称、成分（如"100%棉"、"涤纶65%/棉35%"等）、克重（g/m²）、幅宽（cm）、工艺（如平纹、斜纹、缎纹等）、颜色、价格（元/米）、最小起订量、交货周期（天）、库存数量（米）等。同时支持上传面料图片，最多可上传5张图片。系统对录入数据进行格式校验，确保参数的标准化和规范性。

（2）多条件筛选查询：采购方可以通过成分、工艺、颜色、价格范围（最低价-最高价）、克重范围（最轻-最重）等多个条件进行组合筛选。系统支持分页加载，每页默认显示20条记录，支持上拉加载更多和下拉刷新。

（3）面料参数对比：采购方可以选择最多5个面料加入对比列表，系统以表格形式并排展示各面料的参数信息，方便用户直观比较不同面料的差异。

（4）面料收藏管理：采购方可以将感兴趣的面料加入收藏夹，系统通过用户ID和面料ID的唯一约束确保同一用户不会重复收藏同一面料。收藏夹页面展示所有已收藏的面料列表，支持取消收藏操作。

（5）面料信息管理：供应商可以查看自己发布的所有面料列表，对面料信息进行编辑修改，以及执行上架/下架操作。下架的面料不会出现在采购方的查询结果中。

### 4.2.2 供需精准匹配模块

供需精准匹配模块是本系统的核心创新模块，通过智能匹配算法实现采购需求与面料资源的自动对接。该模块的功能设计如下：

（1）需求发布：采购方填写采购需求信息，包括需求标题、成分要求、克重范围（最小值-最大值）、幅宽范围（最小值-最大值）、工艺要求、颜色要求、价格区间（最低价-最高价）和采购数量。需求发布后状态设为"open"（开放）。

（2）自动匹配触发：系统在两种场景下自动触发匹配引擎——新需求发布时，将该需求与所有状态为"active"的面料进行匹配；新面料上架时，将该面料与所有状态为"open"的需求进行匹配。

（3）匹配结果展示：匹配结果按综合得分降序排列，展示每个匹配面料的综合匹配度（0-100分）以及各维度的分项得分（成分得分、工艺得分、克重得分、价格得分、幅宽得分），帮助采购方全面了解匹配情况。

（4）报价管理：供应商可以针对采购方的需求提交报价，填写报价价格（元/米）、交货周期（天）和附加说明。采购方可以查看所有报价并选择接受某个报价，接受报价后系统自动创建订单。

### 4.2.3 样品与订单协同模块

样品与订单协同模块负责打通从样品申请到订单完成的全流程，实现业务流程的闭环管理。该模块的功能设计如下：

（1）样品申请与审核：采购方可以针对感兴趣的面料向供应商申请样品，填写样品数量和收货地址。供应商收到申请后进行审核，可以批准或拒绝申请，拒绝时需填写拒绝原因。

（2）样品物流追踪：供应商批准样品申请并寄出后，填写物流单号和物流信息。采购方可以实时查看样品的物流状态，确认收到样品后更新状态。样品状态流转为：pending（待审核）→ approved/rejected（已批准/已拒绝）→ shipping（运输中）→ received（已收货）。

（3）订单创建与管理：采购方接受供应商报价后，系统自动创建订单，生成唯一订单号。订单包含订单行项（面料信息、数量、单价、小计）、总金额、收货地址等信息。订单状态流转为：pending（待确认）→ confirmed（已确认）→ producing（生产中）→ shipped（已发货）→ received（已收货）→ completed（已完成）。

（4）订单状态流转：供应商可以依次更新订单状态（确认订单→开始生产→发货），发货时填写物流追踪号。采购方可以确认收货，完成订单闭环。系统在每次状态变更时自动发送通知消息。



### 4.2.4 消息通知模块

消息通知模块负责系统内的信息传递和用户间的实时沟通，包括系统通知和即时对话两个子功能。

（1）系统通知：系统在关键业务事件发生时自动向相关用户推送通知消息。通知类型包括：match（匹配结果通知）、quote（报价通知）、order（订单状态变更通知）、logistics（物流更新通知）、review（审核结果通知）和system（系统公告）。每条通知包含标题、内容、关联业务ID和类型，支持已读/未读状态管理。

（2）即时对话：采购方和供应商之间可以围绕某个采购需求建立对话，进行一对一的文字沟通。对话以需求为纽带，同一需求下同一对买卖双方只能建立一个对话。系统记录每条聊天消息的发送者、内容、类型（文本消息/系统消息）和已读状态，支持消息的分页加载。

（3）未读消息管理：系统维护每个用户的未读消息计数，包括未读对话消息数和未读通知消息数。前端通过定时轮询（每60秒）获取最新的未读计数，并在底部导航栏的消息图标上显示未读徽章。

### 4.2.5 用户认证与权限模块

用户认证与权限模块负责用户的注册、登录、身份验证和权限控制。

（1）用户注册：支持手机号+验证码的注册方式，用户注册时需选择角色（采购方buyer或供应商supplier），填写企业名称、联系人姓名等基本信息。密码采用Werkzeug库的generate_password_hash函数进行哈希加密存储。

（2）用户登录：支持两种登录方式——手机号+密码登录和微信授权登录。微信授权登录通过wx.login获取临时code，后端调用微信接口换取用户openid，实现免密登录。登录成功后服务端生成JWT令牌返回给前端。

（3）角色权限控制：系统定义了三种用户角色（buyer、supplier、admin），不同角色具有不同的API访问权限。后端通过自定义装饰器@role_required()实现角色级别的权限校验，通过@certification_required()装饰器确保用户资质认证通过后才能执行关键业务操作（如发布面料、发布需求等）。

（4）资质认证：供应商和采购方注册后需要提交资质认证申请，管理员审核通过后才能使用平台的核心功能。认证状态包括：pending（待审核）、approved（已通过）和rejected（已拒绝）。

## 4.5 智能匹配算法设计

智能匹配算法是本系统的核心创新点，基于AHP层次分析法实现采购需求与面料资源的多维度加权评分匹配。本节详细阐述算法的设计思路和计算方法。

### 4.5.1 算法总体框架

匹配算法的总体框架分为三个步骤：维度评分、加权聚合和结果筛选。

第一步，维度评分：对每一对供需组合，分别从成分、工艺、克重、价格、幅宽五个维度计算单维度匹配得分，每个维度的得分范围为0-100分。

第二步，加权聚合：根据AHP方法确定的权重向量，对五个维度的得分进行加权求和，计算综合匹配得分。计算公式如下：

Score = W_composition × S_composition + W_craft × S_craft + W_weight × S_weight + W_price × S_price + W_width × S_width

其中，W为各维度权重，S为各维度得分。根据行业专家访谈和文献调研确定的权重分配为：

W_composition = 0.30（成分权重）
W_craft = 0.25（工艺权重）
W_weight = 0.20（克重权重）
W_price = 0.15（价格权重）
W_width = 0.10（幅宽权重）

第三步，结果筛选：将综合得分大于设定阈值的匹配结果保存至数据库，按得分降序排列供用户查看。

### 4.5.2 维度评分规则

根据参数类型的不同，本系统设计了两种评分方法：文本相似度评分和数值范围评分。

（1）文本相似度评分

文本相似度评分用于成分和工艺两个维度，评分规则如下：

- 若需求未指定该参数（为空），则得分为100分，表示无约束；
- 若面料参数与需求参数完全匹配（字符串相等），则得分为100分；
- 若面料参数与需求参数存在部分词汇重叠，则根据重叠比例计算得分，得分范围为50-80分；
- 对于中文文本，系统还支持字符级别的匹配，计算共同字符占比，得分范围为50-80分；
- 若完全不匹配，则得分为0分。

文本相似度评分的计算过程：首先将需求参数和面料参数进行分词处理（按空格和常见分隔符拆分），然后计算两个词集合的交集与并集之比（类似Jaccard相似系数），最后将相似度映射到50-80分的区间。

（2）数值范围评分

数值范围评分用于克重、价格和幅宽三个维度，评分规则如下：

- 若需求未指定范围约束（最小值和最大值均为空），则得分为100分，表示无约束；
- 若面料参数值落在需求指定的范围内（min <= value <= max），则得分为100分；
- 若面料参数值超出范围但在20%容差范围内，则按距离线性衰减计算得分；
- 若面料参数值超出20%容差范围，则得分为0分。

数值范围评分的容差机制设计考虑了纺织行业的实际情况——面料参数在生产过程中存在一定的波动范围，严格的精确匹配可能导致大量合理的面料被排除。20%的容差阈值是根据行业惯例和专家建议确定的。

### 4.5.3 匹配触发机制

系统设计了双向自动匹配触发机制：

（1）需求驱动匹配：当采购方发布新需求时，系统自动将该需求与数据库中所有状态为"active"的面料进行匹配计算，将匹配结果保存至match_result表，并向采购方发送匹配结果通知。

（2）面料驱动匹配：当供应商发布新面料（或将面料状态更新为"active"）时，系统自动将该面料与数据库中所有状态为"open"的需求进行匹配计算，将匹配结果保存至match_result表，并向相关采购方发送新匹配通知。

这种双向触发机制确保了供需信息的实时对接，无论是新需求还是新面料，都能在第一时间找到潜在的匹配对象。

### 5.2.4 角色权限装饰器

系统通过自定义装饰器实现角色级别的权限控制，确保不同角色只能访问其权限范围内的API接口。

```python
# server/routes/auth.py
from functools import wraps
from flask_jwt_extended import get_jwt_identity

def role_required(*roles):
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            user_id = get_jwt_identity()
            user = User.query.get(int(user_id))
            if not user or user.role not in roles:
                return jsonify({"error": "权限不足"}), 403
            return fn(*args, **kwargs)
        return wrapper
    return decorator
```

```python
def get_fabrics():
    query = Fabric.query.filter_by(status="active")

    composition = request.args.get("composition")

    page = request.args.get("page", 1, type=int)
    per_page = request.args.get("per_page", 20, type=int)
    pagination = query.order_by(Fabric.created_at.desc()).paginate(
        page=page, per_page=per_page, error_out=False
    )

    return jsonify({
```

## 5.4 供需匹配模块实现

### 5.4.1 匹配引擎核心类

供需匹配引擎是本系统的核心算法模块，封装在MatchingEngine类中。该类实现了基于AHP权重的五维评分匹配算法，包含文本相似度评分和数值范围评分两种评分方法。

```python
# server/services/matching.py
class MatchingEngine:
    DEFAULT_WEIGHTS = {
        "composition": 0.30,
        "weight": 0.20,
        "craft": 0.25,
        "price": 0.15,
        "width": 0.10
    }

    def match(self, demand, fabrics, weights=None):
        if weights is None:
            weights = self.DEFAULT_WEIGHTS
        results = []
        for fabric in fabrics:
            score_detail = self._calculate_scores(demand, fabric)
            total = sum(weights[k] * score_detail[k] for k in weights)
            results.append({
                "fabric_id": fabric.id,
                "score": round(total, 2),
                "score_detail": score_detail
            })
        results.sort(key=lambda x: x["score"], reverse=True)
        return results
```

### 5.4.2 文本相似度评分实现

文本相似度评分用于成分和工艺两个维度的匹配计算。算法首先进行精确匹配判断，然后通过词汇重叠度和字符级别匹配计算相似度得分。

```python
    def _text_score(self, demand_text, fabric_text):
        if not demand_text:
            return 100  # 无约束
        if not fabric_text:
            return 0

        if demand_text.strip().lower() == fabric_text.strip().lower():
            return 100  # 完全匹配

        # 词汇重叠度计算
        demand_words = set(demand_text.lower().split())
        fabric_words = set(fabric_text.lower().split())
        if demand_words and fabric_words:
            overlap = len(demand_words & fabric_words)
            total = len(demand_words | fabric_words)
            if overlap > 0:
                ratio = overlap / total
                return 50 + ratio * 30

        # 中文字符级别匹配
        common = set(demand_text) & set(fabric_text)
        if common:
            ratio = len(common) / max(len(set(demand_text)), len(set(fabric_text)))
            return 50 + ratio * 30

        return 0
```

### 5.4.3 数值范围评分实现

数值范围评分用于克重、价格和幅宽三个维度的匹配计算。算法支持20%的容差范围，在容差范围内按距离线性衰减计算得分。

```python
    def _numeric_range_score(self, value, range_min, range_max):
        if range_min is None and range_max is None:
            return 100  # 无约束

        if range_min is not None and range_max is not None:
            if range_min <= value <= range_max:
                return 100  # 在范围内

            span = range_max - range_min
            tolerance = span * 0.2 if span > 0 else max(abs(range_min), abs(range_max)) * 0.2

            if value < range_min:
                distance = range_min - value
            else:
                distance = value - range_max

            if distance <= tolerance and tolerance > 0:
                return 100 * (1 - distance / tolerance)

        elif range_min is not None:
            if value >= range_min:
                return 100
            tolerance = abs(range_min) * 0.2
            distance = range_min - value
            if distance <= tolerance and tolerance > 0:
                return 100 * (1 - distance / tolerance)

        elif range_max is not None:
            if value <= range_max:
                return 100
            tolerance = abs(range_max) * 0.2
            distance = value - range_max
            if distance <= tolerance and tolerance > 0:
                return 100 * (1 - distance / tolerance)

        return 0
```

### 5.8.1 网络请求封装

前端通过request.js工具模块对微信小程序的wx.request接口进行了统一封装，实现了以下功能：自动注入JWT令牌到请求头、统一的错误处理和提示、401响应自动清除登录态并跳转登录页、网络异常时的重试提示。

```javascript
// miniprogram/utils/request.js
var auth = require("./auth");
var env = require("../config/env");

var baseUrl = env.getBaseUrl();

function request(url, method, data) {
  return new Promise(function(resolve, reject) {
    var header = { "Content-Type": "application/json" };
    var token = auth.getToken();
    if (token) {
      header["Authorization"] = "Bearer " + token;
    }

    wx.request({
      url: baseUrl + url,
      method: method,
      data: data,
      header: header,
      success: function(res) {
        if (res.statusCode === 401) {
          auth.clearToken();
          wx.navigateTo({ url: "/pages/login/login" });
          reject(new Error("未登录"));
          return;
        }
        if (res.statusCode >= 400) {
          var errMsg = (res.data && res.data.error) || "请求失败";
          wx.showToast({ title: errMsg, icon: "none" });
          reject(new Error(errMsg));
          return;
        }
        resolve(res.data);
      },
      fail: function(err) {
        wx.showModal({
          title: "网络异常",
          content: "请检查网络连接后重试",
          showCancel: false
        });
        reject(err);
      }
    });
  });
}

module.exports = {
  get: function(url, data) { return request(url, "GET", data); },
  post: function(url, data) { return request(url, "POST", data); },
  put: function(url, data) { return request(url, "PUT", data); },
  del: function(url, data) { return request(url, "DELETE", data); }
};
```

应用启动时，系统自动检查本地存储中的登录态信息，如果存在有效的JWT令牌则恢复登录状态，避免用户每次打开小程序都需要重新登录。同时启动未读消息轮询，确保消息徽章的实时更新。

```javascript
// miniprogram/app.js (onLaunch)
App({
  onLaunch: function() {
    this.checkLoginStatus();
  },

  checkLoginStatus: function() {
    var token = wx.getStorageSync("token");
    var userInfo = wx.getStorageSync("userInfo");
    if (token && userInfo) {
      this.globalData.token = token;
      this.globalData.userInfo = userInfo;
      this.globalData.isLoggedIn = true;
    }
  },

  globalData: {
    userInfo: null,
    token: "",
    isLoggedIn: false,
    unreadCount: 0,
    platform: {
      name: "纺织面料平台",
      version: "1.0.0"
    }
  }
});
```

# 第7章 总结与展望

## 7.1 工作总结

本文围绕纺织面料行业信息化程度低、供需对接效率不高的现实问题，设计并实现了一个基于微信小程序的纺织面料智能查询与供需对接平台。论文的主要工作和成果如下：

（1）完成了系统的需求分析。通过对纺织面料行业的调研，明确了采购方、供应商和管理员三类用户角色的功能需求，梳理了面料查询、供需匹配、样品管理、订单协同和消息通知五大核心业务流程，并从性能、安全性和可用性三个方面提出了非功能性需求。

（2）完成了系统的架构设计和详细设计。采用前后端分离的B/S架构，前端使用微信小程序原生框架，后端使用Python Flask框架，数据库使用MySQL。设计了合理的数据库表结构，定义了规范的RESTful API接口，并采用模块化的系统架构，将系统划分为用户认证、面料管理、需求管理、匹配引擎、订单管理、样品管理、消息通知和管理后台八个功能模块。

（3）实现了基于AHP权重的供需智能匹配算法。该算法从成分、克重、工艺、价格和幅宽五个维度对面料与需求进行综合评分匹配，采用文本相似度和数值范围评分两种方法，支持20%的容差范围和线性衰减计算，能够有效地为采购方推荐最匹配的面料资源。

（4）实现了完整的业务闭环流程。从需求发布、智能匹配、供应商报价、样品申请与审核、订单创建与状态流转，到消息通知与即时沟通，形成了完整的供需对接业务链条。系统通过状态机模式管理样品和订单的生命周期，通过事件驱动的通知机制保证信息的及时传达。

（5）完成了系统的测试验证。设计了30个功能测试用例，覆盖了系统的核心功能模块，所有用例均测试通过。性能测试结果表明，关键接口响应时间均在1秒以内，匹配算法在5000条面料规模下执行时间在2秒以内，满足系统的性能需求。

## 7.2 不足与展望

尽管本系统已经实现了预期的功能目标，但仍存在一些不足之处，有待在后续工作中进一步改进和完善：

（1）匹配算法的智能化程度有待提升。当前的匹配算法基于固定的AHP权重进行评分计算，尚未引入机器学习技术。未来可以考虑引入协同过滤推荐算法，根据用户的历史采购行为和偏好进行个性化推荐；也可以引入自然语言处理技术，支持用户通过自然语言描述需求进行智能匹配。

（2）图像识别功能尚未实现。纺织面料的外观特征（如纹理、花型、光泽度等）是采购决策的重要参考因素。未来可以引入计算机视觉技术，实现面料图片的智能识别和相似面料推荐，进一步提升查询效率。

（3）实时通信能力有待增强。当前的消息系统采用轮询机制获取新消息，存在一定的延迟。未来可以引入WebSocket长连接技术，实现消息的实时推送，提升沟通效率和用户体验。

（4）数据分析能力有待加强。当前管理后台仅提供基础的统计数据。未来可以引入数据可视化和商业智能分析功能，为平台运营提供更深入的数据洞察，如面料价格趋势分析、供需热度分析、用户行为分析等。

（5）系统的可扩展性有待优化。随着用户量和数据量的增长，单体架构可能面临性能瓶颈。未来可以考虑将系统重构为微服务架构，引入消息队列、缓存中间件和分布式数据库等技术，提升系统的并发处理能力和可扩展性。

（6）移动端体验有待丰富。当前系统仅支持微信小程序端，未来可以考虑开发H5版本和APP版本，覆盖更多的使用场景和用户群体。同时可以引入AR（增强现实）技术，让用户通过手机摄像头直观感受面料的实际效果。

综上所述，本系统为纺织面料行业的信息化和智能化提供了一个可行的解决方案，在面料查询和供需对接方面取得了一定的成果。随着人工智能、大数据和物联网等技术的不断发展，纺织面料行业的数字化转型将迎来更广阔的发展空间，本系统也将在后续工作中持续迭代优化，为行业发展贡献更大的价值。

## 【合并至5.3节后】5.3.5 面料对比功能实现

面料对比是本系统的特色功能之一，允许用户选择最多5个面料进行参数横向对比。后端通过批量查询接口一次性获取多个面料的详细信息，前端以表格形式展示各面料的关键参数差异。

```python
# server/routes/fabric.py
@fabric_bp.route("/api/fabrics/compare", methods=["GET"])
def compare_fabrics():
    ids_str = request.args.get("ids", "")
    if not ids_str:
        return jsonify({"error": "请选择要对比的面料"}), 400

    ids = [int(i) for i in ids_str.split(",") if i.strip().isdigit()]
    if len(ids) < 2:
        return jsonify({"error": "至少选择2个面料进行对比"}), 400
    if len(ids) > 5:
        return jsonify({"error": "最多对比5个面料"}), 400

    fabrics = Fabric.query.filter(Fabric.id.in_(ids)).all()
    return jsonify({
        "fabrics": [fabric_to_dict(f) for f in fabrics]
    })
```

前端对比页面采用横向滚动表格布局，将面料的名称、成分、克重、幅宽、工艺、颜色、价格、起订量、交货周期等参数逐行对比展示。对于数值型参数，系统自动高亮最优值（如最低价格、最短交货周期），帮助采购方快速识别最具优势的面料。

需求发布接口对采购方提交的需求数据进行严格的服务端验证，确保数据的完整性和合理性。验证规则包括：标题不能为空且长度不超过200字符、数值范围的上限必须大于下限、价格和数量必须为正数等。

## 【合并至5.6节后】5.6.4 收藏功能实现

```python
@fabric_bp.route("/api/fabrics/<int:fabric_id>/favorite", methods=["POST"])
@jwt_required()
def toggle_favorite(fabric_id):
    user_id = int(get_jwt_identity())
    fabric = Fabric.query.get_or_404(fabric_id)

    existing = Favorite.query.filter_by(
        user_id=user_id, fabric_id=fabric_id

    if existing:

        db.session.commit()

    else:
        fav = Favorite(user_id=user_id, fabric_id=fabric_id)
```

## 【合并至3.2节后】补充：系统用例分析

为更清晰地描述系统的功能需求，本节对三类用户角色的核心用例进行详细分析。

（1）采购方用例分析

采购方是平台的核心用户群体，其主要使用场景包括：注册登录后完善企业信息并提交资质认证；通过多条件筛选查询面料库，浏览面料详情并收藏感兴趣的面料；选择多个面料进行参数横向对比，辅助采购决策；发布采购需求，系统自动匹配推荐面料；查看匹配结果，向供应商发起沟通或申请样品；接受供应商报价后系统自动生成订单，跟踪订单状态直至完成交易。采购方的核心诉求是高效找到符合要求的面料资源，降低采购成本和沟通成本。

（2）供应商用例分析

供应商是平台的面料资源提供方，其主要使用场景包括：注册登录后完善企业信息和资质材料；发布和管理面料产品信息，包括面料的基本参数、图片、价格和库存等；接收系统推送的匹配需求通知，主动查看与自身面料匹配的采购需求；与采购方进行在线沟通，了解具体需求细节；审核采购方的样品申请，安排样品寄送；对采购需求提交报价，跟踪订单执行进度。供应商的核心诉求是拓展销售渠道，精准对接有效客户，提升面料的销售转化率。

（3）管理员用例分析

管理员负责平台的日常运营和管理，其主要使用场景包括：审核用户提交的企业资质认证申请，确保平台用户的真实性和可信度；查看平台运营数据统计，包括用户总数、面料总数、需求总数、订单总数等关键指标；管理平台内容，处理违规信息和用户投诉。管理员的核心诉求是维护平台的健康运营环境，保障交易的安全性和规范性。

## 【合并至4.3节后】补充：API接口安全设计

系统在API接口层面采取了多重安全防护措施，确保数据传输和访问的安全性：

（1）身份认证机制。所有需要登录才能访问的接口均通过@jwt_required()装饰器进行保护，未携带有效JWT令牌的请求将被拒绝并返回401状态码。JWT令牌设置了合理的过期时间，过期后需要重新登录获取新令牌。

（2）角色权限控制。通过自定义的@role_required()装饰器实现细粒度的角色权限控制。例如，面料发布接口仅允许supplier角色访问，需求发布接口仅允许buyer角色访问，管理后台接口仅允许admin角色访问。权限不足的请求返回403状态码。

（3）数据归属校验。除了角色权限控制外，系统还对数据的归属关系进行校验。例如，供应商只能修改和删除自己发布的面料，采购方只能操作自己创建的需求和订单。这种双重校验机制有效防止了越权操作。

（4）输入数据验证。所有API接口对输入数据进行严格的服务端验证，包括必填字段检查、数据类型校验、数值范围校验和字符串长度限制等。服务端验证是安全防线的最后一道关卡，即使前端验证被绕过，服务端仍能拦截非法数据。

（5）密码安全存储。用户密码使用Werkzeug库提供的pbkdf2:sha256算法进行哈希加密存储，该算法通过多轮迭代和随机盐值确保密码哈希的安全性，即使数据库泄露也无法还原明文密码。

（6）跨域资源共享控制。后端通过Flask-CORS扩展配置了跨域访问策略，仅允许来自微信小程序域名的请求访问API接口，防止未授权的第三方应用调用系统接口。
